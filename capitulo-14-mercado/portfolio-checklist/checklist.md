# Checklist de Portfolio — Capítulo 14

## O que um portfolio moderno precisa demonstrar em 2026

### Repositório GitHub
- [ ] README profissional com badges, instruções claras e demo (GIF ou link)
- [ ] Código com TypeScript (não apenas JavaScript)
- [ ] Testes — pelo menos unitários com Vitest
- [ ] CI/CD configurado (GitHub Actions rodando lint + testes)
- [ ] Commits semânticos: `feat:`, `fix:`, `refactor:`, `test:`
- [ ] `.gitignore` correto — sem `node_modules` ou `.env` commitados

### Stack que aparece nas vagas em 2026
- [ ] React ou Vue com TypeScript
- [ ] Next.js (App Router) ou Remix — pelo menos um projeto
- [ ] Vite como build tool
- [ ] ESLint + Prettier configurados
- [ ] Pelo menos um projeto com testes E2E (Playwright)

### Para vagas de backend/fullstack
- [ ] Node.js com TypeScript e ESM
- [ ] REST API com validação (Zod)
- [ ] Pelo menos um projeto com WebSockets ou SSE
- [ ] Docker/docker-compose no repositório
- [ ] Variáveis de ambiente documentadas (.env.example)

### Demonstre que você sabe trabalhar em time
- [ ] Issues abertas e fechadas no repositório
- [ ] Pull Requests com descrição — mesmo que seja você mesmo
- [ ] CONTRIBUTING.md explicando como rodar o projeto

## Checklist de entrevista técnica

### Perguntas que você deve saber responder
- [ ] "Qual a diferença entre ESM e CommonJS? Por que isso importa?"
- [ ] "Explique o event loop do JavaScript"
- [ ] "Quando você usaria GraphQL em vez de REST?"
- [ ] "O que é hidratação em SSR e por que importa?"
- [ ] "Como você migraria um projeto legado sem parar a produção?"
- [ ] "Me conta sobre uma decisão técnica difícil que você tomou"

### Red flags que você deve evitar
- [ ] Não saber explicar o código que escreveu com IA
- [ ] "Funciona na minha máquina" sem reprodução documentada
- [ ] Repositório público com tokens ou senhas expostos
- [ ] Commits com mensagens "fix", "update", "ajuste"
